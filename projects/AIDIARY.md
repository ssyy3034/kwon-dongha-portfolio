# Aidiary - AI 기반 산모 감정 일기 서비스

> 캡스톤 디자인 (2025.04~06) → 수료 후 개인 개선 (2026.01~02) / 2인 개발 (기획, 풀스택 개발)

산모가 일기를 쓰면 AI가 감정을 분석해 피드백을 제공하고, 부모 사진으로 아기 캐릭터를 생성하는 서비스입니다. MediaPipe 등 Python 전용 라이브러리 활용이 필요하여 Spring Boot + Flask 이중 서버 구조로 분리했습니다.

## Tech Stack

| 영역         | 기술 스택                              |
| :----------- | :------------------------------------- |
| **Frontend** | React, TypeScript                      |
| **Backend**  | Spring Boot, Flask, RabbitMQ, Caffeine |
| **AI/ML**    | MediaPipe, LLM API                     |
| **Infra**    | AWS (EC2, S3, CloudFront), Docker Compose |

## 담당 구현

### Backend (기여도 80%)

- Spring Security + JWT 인증/인가 및 산모 일기 CRUD API 구축
- MediaPipe·LLM 연동 태아 캐릭터 합성 파이프라인 — Python 전용 ML 라이브러리를 위해 Spring Boot + Flask 이중 서버로 분리하고, RabbitMQ로 비동기 처리
- 사용자 데이터 기반 개인화 정보 제공 — 임신 주차·건강 정보를 컨텍스트에 반영하고, Caffeine 캐싱으로 반복 API 호출 절감

### Infra (기여도 100%)

- Docker Compose로 Spring Boot·Flask·RabbitMQ 등 5개 서비스 오케스트레이션 및 AWS EC2 배포
- 프론트 S3 + CloudFront CDN, 백엔드 EC2 배포, GitHub Actions CI/CD 파이프라인 구축

### Frontend (기여도 20%)

- 부모 사진 업로드 → AI 합성 → 결과 표시까지의 비동기 렌더링 흐름 구현 — 이미지 압축을 Web Worker로 분리하여 모바일 UI 블로킹 방지

## Troubleshooting

### 1. AI 이미지 합성 동기 블로킹 → RabbitMQ 비동기 전환

**문제**

ML 이미지 합성(~30초) 작업이 톰캣 스레드를 직접 점유하여, 동시 사용자 10~20명 수준에서도 전체 서비스가 마비되었습니다(k6 실측 1.16 TPS).

**접근 및 한계**

@Async로 톰캣 스레드를 즉시 반환하도록 전환하여 처리량을 개선했으나, 부하 테스트에서 요청이 몰리자 로컬 큐 포화로 메모리 고갈이 발생했습니다. AI 합성 작업의 무거운 특성상 별도 서비스 분리를 검토하는 과정에서, WAS 내부 메모리 큐에 의존하는 @Async는 서버 종료·재배포 시 작업이 유실되는 구조적 한계도 확인했습니다.

**액션**

RabbitMQ를 도입하여 메시지 영속성을 확보하고, AI 합성 처리를 WAS와 물리적으로 분리했습니다.

**결과**

WAS 처리량 1.16 → 1,949 TPS. 동기 대기 30초 → 비동기 전환으로 즉시 응답(4.9ms). 500 VU에서 p95 318ms. AI 서비스가 다운되더라도 로그인·일기 작성 등 주요 기능은 정상 동작하는 장애 격리 구조를 확보했습니다.

### 2. Gemini API 매 요청 호출 → 컨텍스트 해시 기반 캐시로 호출 최소화

**문제**

주차별 정보와 오늘의 질문 기능이 매 요청마다 Gemini API를 호출하고 있어, 응답 시간 평균 487ms에 API 비용이 요청 수에 비례해 선형 증가하는 구조였습니다. 주차별 정보는 개인화도 되지 않아 모든 산모에게 동일한 응답을 반환하고 있었습니다.

**접근**

같은 주차·같은 건강 상태의 산모에게는 동일한 응답이 돌아오므로, 상태가 바뀌지 않은 요청은 캐싱할 수 있다고 판단했습니다. 그러나 상태가 불규칙하게 바뀌므로 고정 TTL로는 무효화 시점을 제어할 수 없었고, 원시 값을 그대로 SHA-256으로 해싱하면 혈압이 1만 변해도 새 키가 생겨 캐시 히트율이 급락했습니다. 이에 감정(긍정/중립/부정)·혈압(정상/주의/위험)·태동(활발/보통/저조) 등 도메인 기준으로 카테고리화한 뒤 해싱하여, 의미 있는 상태 변화에서만 캐시가 무효화되도록 설계했습니다.

오늘의 질문은 하루 1회 생성되는 공통 데이터이므로 DB에 날짜 기반으로 저장하여 분리했습니다.

**결과**

주차별 정보 응답 487ms → 0.1ms(Caffeine HIT), 오늘의 질문은 Gemini 호출 일 1회 고정. 전체 Gemini 호출이 매 요청에서 컨텍스트 변경 시 또는 일 1회로 고정되어 API 비용을 대폭 절감했습니다.
