# StoLink & StoRead - 웹소설 작가를 위한 지식 관리 플랫폼

> 크래프톤 정글 최종 프로젝트 / 5인 개발 / 기획, 풀스택 개발 | 2025.12 — 2026.02

웹소설 작가가 글을 쓰면 AI가 인물·관계·사건을 자동 분석하여 그래프로 시각화해주는 창작 보조 플랫폼입니다. 작가는 StoLink에서 에디터와 데이터 시각화를 통해 체계적으로 글을 쓰고, 배포한 글을 통해 StoRead에서 독자들이 관계도와 함께 작품을 열람하거나 구매할 수 있습니다.

## Frontend Troubleshooting

### 1. 렌더링 성능 최적화: SVG → Canvas 전환

**문제**

D3.js SVG 기반 소설 인물 관계도에서 노드 50개만 넘어도 클릭 및 드래그 응답이 420ms까지 느려지는 문제가 발생했습니다.

**원인 분석**

DevTools Performance 탭으로 분석한 결과, 매 프레임마다 수백 개 SVG DOM 요소의 스타일 재계산(Style Recalculation) → 레이아웃 재배치(Reflow) → 다시 그리기(Paint)가 연쇄적으로 발생하며 메인 스레드를 블로킹하는 것이 병목임을 확인했습니다.

**액션**

SVG/DOM의 한계를 이해하고 Canvas 렌더링으로 전환했습니다.

- 수천 개의 DOM 요소를 단일 Canvas 엘리먼트로 통합하여 브라우저 렌더링 부하를 제거
- 캐릭터 이미지는 Offscreen Canvas에 비트맵 캐싱하고, 개별 노드를 단일 렌더 루프에서 일괄 렌더링
- Canvas에서는 DOM 이벤트를 사용할 수 없으므로, 좌표 기반 히트 테스트로 클릭/드래그 이벤트를 직접 구현

**결과**

- 650+ 노드에서도 60FPS 유지
- INP 420ms → 64ms로 개선

---

### 2. 데이터 처리 최적화: 트리 변환 메모이제이션

**문제**

폴더 구조가 깊어질수록 사이드바의 렌더링 속도가 느려졌습니다. 계층형 문서 구조(Flat Array → Nested Tree)를 매 렌더링마다 재귀적으로 변환하고 있어 O(n²) 연산이 반복 발생, 문서 100개 기준 15~20ms가 소요되었습니다.

**액션**

- HashMap 기반 O(n) 트리 조립 로직으로 리팩토링
- `useMemo`로 데이터 변경 시에만 재연산되도록 최적화
- React의 참조 동등성을 활용해 불필요한 재연산을 제거

**결과**

- O(n²) → O(n) 전환: 문서 100개 기준 15~20ms → 1ms 이하로 개선

---

### 3. 번들 최적화

**문제**

초기 로딩이 느리다는 피드백을 받았습니다. Lighthouse 분석 결과, Export 라이브러리(docx 714KB, jspdf 341KB)가 초기 번들에 포함되어 있었습니다.

**액션**

- `vite.config.ts`의 manualChunks 설정으로 용도별 청크 분리
- 마우스 호버 시 사전 로딩(Prefetch) 적용하여 사용자가 실제로 해당 기능에 진입할 때는 이미 리소스가 준비되도록 구성

**결과**

- 초기 JS 번들 450KB → 187KB (58% 절감, gzip)
- 라이브러리별 청크 분리로 캐시 효율 증대

## Backend Troubleshooting

### 1. 서버 다운으로 인한 서비스 장애 해결 - 원인 파악 후 코드 개선

**문제**

크롬 개발자 도구에서 API 요청이 타임아웃으로 실패하는 것을 확인했으나, 서버의 CPU·메모리는 정상이었습니다. 쓰레드 덤프를 떠서 분석한 결과, 다수의 톰캣 쓰레드가 AI 이미지 서버 헬스체크용 `RestTemplate.getForObject()`에서 Socket Read(RUNNABLE) 상태로 블로킹되어 있었습니다.

**원인 분석**

추적해보니 해당 RestTemplate의 Timeout을 `@Value` 필드 주입으로 설정했으나, 빈 라이프사이클상 필드 주입은 생성자 이후에 이루어지므로 RestTemplate 생성 시점에는 값이 0인 채로 남아 기본값(무한 대기)으로 동작하고 있었습니다. 여기에 이 호출이 `@TransactionalEventListener(REQUIRES_NEW)` 내부에서 실행되고 있어, 외부 서버 무응답 시 DB 커넥션을 점유한 채 무한 대기 → HikariCP 풀 고갈 → 후속 요청이 커넥션을 획득하지 못하고 WAITING 상태로 전이되는 Cascading Failure 구조였습니다.

**액션**

필드 주입이 빈 라이프사이클 문제의 근본 원인임을 파악하고, 해당 서비스의 설정 주입 방식을 생성자 주입으로 리팩토링하여 빈 생성 시점에 Timeout이 확실히 반영되도록 수정했습니다.

**결과 및 교훈**

근본적인 원인은 트랜잭션 내부에서 네트워크 I/O를 수행하는 아키텍처적 결함이었습니다. 해당 코드는 로직이 강하게 결합되어 있어 즉시 리팩토링하기 어려웠지만, 이 경험을 교훈 삼아 이후 결제 PG 연동 설계에서는 트랜잭션 외부에서 PG사 통신을 완료한 뒤 결과 DB 업데이트에만 최소한의 트랜잭션을 여는 구조를 선제적으로 적용했습니다.

### 2. PG 연동 커넥션 풀 고갈 방지 — I/O-트랜잭션 분리 및 보상 재시도 설계

**문제**

위 Cascading Failure를 겪은 뒤, 결제 플로우에도 동일한 구조적 위험이 있음을 인지했습니다. 결제 승인 로직이 하나의 트랜잭션 안에서 토스페이먼츠 API를 동기 호출하고 있었고, HikariCP 기본 풀이 10개인 상황에서 PG 응답이 1~2초 소요되므로 동시 결제 10건만으로도 일반 API까지 커넥션 부족이 전파될 수 있는 구조였습니다.

**액션**

네트워크 I/O와 DB 트랜잭션을 분리했습니다. 첫 번째 트랜잭션에서 결제 검증과 상태 전환만 수행하고 즉시 커밋하여 커넥션을 반환한 뒤, 트랜잭션 밖에서 PG API를 호출하고, 두 번째 트랜잭션에서 승인 결과만 DB에 반영하는 구조로 수정했습니다. 분리로 인해 발생하는 데이터 불일치는 보상 트랜잭션과 재시도 구조로 최종적 일관성을 확보했습니다.

**결과**

개선 후 PG 응답 지연이 DB 커넥션 풀에 영향을 주지 않는 구조로 전환되었고, 결제 API에 지연이 생겨도 일반 API 정상 동작을 확인했습니다.

### 3. 계층형 문서 조회 N+1 해결 - Fetch Join + In-Memory 트리로 450ms → 25ms

**문제**

에디터 문서 사이드바의 로딩이 체감상 느려, 크롬 개발자 도구의 Network 탭을 확인한 결과 사이드바 트리를 호출하는 API의 응답 시간이 450ms를 차지하고 있었습니다.

**원인 분석**

무거운 비즈니스 로직이 없는 단순 읽기 API였기에 쿼리 로그로 검증한 결과, 문서 조회 시 부모-자식 관계의 Lazy Loading으로 인해 문서마다 부모를 개별 조회하는 N+1 문제가 발생하여 총 N+1개의 쿼리가 실행되고 있었습니다.

**액션**

Fetch Join으로 전체 문서를 1개의 쿼리로 로드한 뒤, 서버에서 HashMap을 활용해 O(N) 시간 복잡도로 트리를 조립하는 방식을 택했습니다.

**결과**

호출 당 쿼리는 N+1개에서 1개로, 응답 속도는 450ms에서 25ms로 18배 개선되었습니다.

### 4. AI 컨텍스트 오염 방지 → 3-Layer 문서 및 멀티 에이전트 파이프라인 구축

**문제**

Claude Code 지침서가 1,000줄을 넘어가면서 단순 텍스트 수정 같은 작업에도 전체 문서가 로딩되어 컨텍스트 윈도우의 약 20%가 규칙 해석에 소모되었고, 레거시 API 명세가 최신 문서와 혼재되어 잘못된 코드가 생성되는 문제가 있었습니다. 또한 협업 과정에서 문서 갱신 누락으로 중복 구현이 발생하는 프로세스 병목이 있었습니다.

**액션**

문서를 3-Layer(Core/Appendix/Spec)로 분리하고, Supervisor 커맨드가 작업 규모(S/M/L)에 맞춰 필요한 layer만 선택적으로 로딩하는 파이프라인을 구축했습니다.

**결과**

- S 사이즈 작업에서는 약 50줄 분량만 로딩하도록 경량화하여 불필요한 토큰 소모를 줄였고, 단순 작업 시 전체 사양서를 거치던 프로세스를 간소화했습니다.
- 레거시 API 명세는 Spec layer에서 분리·아카이빙하여 최신 명세만 참조되도록 했고, 작업 완료 시 Librarian 커맨드가 커밋 기록과 변경 코드를 기반으로 문서를 자동 갱신하여 교대 개발 시 문서 누락을 방지했습니다.

## 5. 팀 협업: Git 워크플로우 자동화

**문제**

5인 팀에서 Git 숙련도 격차로 브랜치 충돌, 커밋 컨벤션 미준수, 이슈 트래킹 누락 등의 문제가 반복 발생했습니다.

**액션**

Antigravity를 활용하여 이슈 생성 → 브랜치 생성 → 커밋 → PR 작성까지 단일 명령으로 연결하는 워크플로우를 구축했습니다.

**결과**

- 개인 도구로 시작했으나 팀 전체가 사용하게 되었고, 이후 다른 프로젝트 팀으로도 확산
- 개발 외적인 프로세스 고민을 덜어내어 팀원들이 핵심 로직 개발에 집중
