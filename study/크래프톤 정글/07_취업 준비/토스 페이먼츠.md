AiDiary 프로젝트에서 조회 API의 응답 지연 문제를 다층 캐싱(Multi-layer Cache) 전략으로 개선한 사례입니다. 초기에는 외부 시스템 대기 시간을 줄이고자 Redis 기반 중앙 캐시를 도입했습니다. 이후 특정 정보(예: 임신 중기 주차)에 조회가 집중되면서, Redis 네트워크 I/O 병목으로 지연이 발생했습니다. 이 오버헤드를 줄이기 위해 애플리케이션 메모리에 Caffeine 본 기반의 L1 로컬 캐시를 구성하고 Redis를 L2 캐시로 활용하는 다층 구조로 전환해 네트워크 홉(Hop)을 최소화했습니다. 구조 변경 후 캐시 아벌란체 현상을 확인했습니다. 일괄 설정된 만료 시간으로 인해 갱신 시점에 대규모 요청이 한 번에 DB로 집중되는 문제였습니다. 이를 방지하기 위해 TTL에 5~15%의 지터(Jitter)를 부여해 부하를 분산시켰습니다. 추가로, 존재하지 않는 파라미터를 반복 요청해 DB 부하를 유발하는 캐시 관통(Penetration)을 막고자 Null 값에 짧은 TTL을 적용하여 캐싱했습니다. 그 결과 평균 응답 시간을 15ms 수준으로 단축하고 불필요한 DB 조회를 제어했습니다.
