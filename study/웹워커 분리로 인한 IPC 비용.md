
### 1. Web Worker와 통신 비용(IPC)의 관계

우리가 자바스크립트에서 **Web Worker**를 만들면, 브라우저는 메인 쓰레드(Main Thread, UI를 그리는 곳)와 완전히 분리된 **새로운 작업 공간(Thread 혹은 Process 수준의 격리)**을 만듭니다.

중요한 것은 이 두 공간은 **메모리를 서로 공유하지 않는다는 점**입니다. 따라서 메인 쓰레드와 워커 쓰레드가 데이터를 주고받으려면 우편물(메시지)을 포장해서 보내고 받는 과정이 필요한데, 이를 브라우저 환경에서는 `postMessage()`를 통해 수행하며, 넓은 의미에서 이를 **프로세스/쓰레드 간 통신(IPC)**이라고 부릅니다.

### 2. 왜 이 통신 과정에서 '오버헤드(병목)'가 생길까요?

메모리를 공유하지 않기 때문에, 메인 쓰레드에서 워커로 데이터를 보낼 때 브라우저는 **"구조화된 복제(Structured Clone)"**라는 작업을 수행합니다.

- **문제 상황 예시:** 사용자가 올린 10MB짜리 고화질 사진의 원본 픽셀 데이터(`ImageData`)를 워커로 그냥 넘긴다고 가정해 봅시다. 브라우저는 메인 쓰레드의 메모리에 있는 10MB 데이터를 **똑같이 복사해서** 워커의 메모리에 새로 찍어내야(Clone) 합니다.
- **결과:** 메모리 사용량이 20MB로 두 배가 되고, 10MB짜리 데이터를 복사하는 시간 동안 **메인 쓰레드가 멈춰버립니다(UI 프리징).** "무거운 작업을 뒤로 빼려고 워커를 썼는데, 정작 데이터를 넘겨주는 통신 과정(IPC)에서 시간이 너무 오래 걸려 UI가 버벅이는 현상"이 발생하는 것입니다.

면접관이 **"오히려 메인 쓰레드에 또 다른 병목을 만들진 않았나요?"**라고 묻는 것은 바로 이 **'복사(통신) 비용'**을 네가 이해하고 워커를 쓴 게 맞냐고 찌르는 것입니다.

### 3. 이것을 어떻게 해결하는가? (방어 전략)

이 무거운 통신(IPC) 비용을 해결하기 위해 사용하는 기술이 앞서 말씀드린 두 가지 방식입니다.

#### 💡 첫 번째 전략: 가벼운 포인터(메타데이터)만 넘기기 **(현재 지원자님 코드의 구현 방식이자 아주 훌륭한 정답!)**

지원자님의 코드(

![](vscode-file://vscode-app/Applications/Antigravity.app/Contents/Resources/app/extensions/theme-symbols/src/icons/files/ts.svg)

useCharacter.ts)를 보면, 이미지의 무거운 픽셀 배열을 직접 넘기지 않고 브라우저가 갖고 있는 사진 파일의 껍데기(메타데이터)인 **`File` 객체**만 넘겨주었습니다. `File` 객체는 실질적인 픽셀 덩어리가 아니라 파일을 가리키는 정보이기 때문에, 복사하는 데 비용이 거의 들지 않습니다. 그리고 워커 내부에서 `createImageBitmap(file)`을 호출해 그제서야 사진의 압축을 풀고 픽셀을 꺼냅니다. 👉 **결론:** 무거운 픽셀을 복사하며 넘긴 게 아니므로 통신 비용 자체를 극적으로 줄였습니다.

#### 🚀 두 번째 전략: Transferable Objects (소유권 이전 - 면접 필살기)

만약 `File` 형태가 아니라 워커에서 이미 픽셀을 다 계산한 무거운 덩어리(`ArrayBuffer`, `ImageBitmap` 등)를 다시 메인 쓰레드로 돌려보내야 할 때는 어떻게 할까요? 이때는 "복사"하지 않고 데이터의 **"소유권 자체를 아예 넘겨버립니다(Zero-Copy)".** 마치 USB 드라이브에 파일을 복사해서 주는 게 아니라, USB 드라이브 자체를 상대방 손에 쥐여주고 내 손을 떼는 것과 같습니다. 이 기술을 **Transferable Objects**라고 부르며, 복사 과정이 0(Zero)이기 때문에 오버헤드가 전혀 발생하지 않습니다.

### 🎙️ 면접관 앞에서 할 수 있는 완벽한 마무리 멘트

> "면접관님이 우려하시는 바와 같이, Web Worker로 데이터를 넘길 때 발생하는 Structured Clone(구조화된 복제) 통신 오버헤드, 즉 IPC 비용을 충분히 인지하고 있었습니다.
> 
> 그래서 메인 스레드에서 이미지를 픽셀 단위로 파싱해 넘기지 않고, **`File` 객체 자체만 `postMessage`로 전달**하여 복사 오버헤드를 최소화했습니다. 사진을 메모리에 올리고 비트맵으로 변환(`createImageBitmap`)하는 진짜 무거운 디코딩 작업은 온전히 Worker 내부에서 실행되도록 위임했기 때문에, 데이터를 넘겨주는 통신 과정에서 메인 스레드가 차단(Block)되는 현상은 발생하지 않았습니다."
