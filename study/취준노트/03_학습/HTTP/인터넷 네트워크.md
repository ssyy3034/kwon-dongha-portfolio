## 웹 브라우저 요청 흐름
DNS 서버 조회. ip랑 포트 정보 조회 후 HTTP 메세지 생성
GET 요청이 갈거고 path, 쿼리 파라미터 전송
패킷 받으면 구글 서버에서 까서 검사 후 응답 데이터 만들어서 보냄.
200ok,컨텐츠 타입, 길이 명시 후 CRLF 하고 메세지
클라이언트(내 화면)가 받아서 렌더링

요즘은 모든것을 HTTP로 전송함.

클라이언트 서버 구조
클라이언트는 서버에 요청을 보내고, 응답을 대기
서버는 요청에 대한 결과를 만들어서 응답.

HTTP의 무상태성
상태를 유지하지 않음.(이전 응답의 문맥을 유지하지 않음)

장점: 확장성 높음. 무한한 서버 증설 가능
단점 : 메세지 양이 커짐. 무상태로 설계할수 없는 경우가 있다.(인증 등)

비연결성
tcp/ip의 경우 기본적으로 연결을 유지한다. 하지만 요청-응답이 끝나면? 연결 종료
서버의 자원을 효율적으로 사용할수 있다.
새로운 요청이 갈때마다 연결을 다시 생성하는데, 사용자가 많아질수록 부하가 커짐.
3way hand-shake 시간이 연결마다 추가. 수많은 자원을 함께 다운로드 하는데, 매 요청마다 연결이 생성됨
http2,3에서는 최적화됨
페이지 하나 다 받을때까지 연결 유지하다가 종료


http3의 경우 udp 사용해서 3way 시간도 줄임.
