# 도메인 이름 (Domain Name)

## 1. 정의

도메인 이름은 **숫자로 된 IP 주소를 사람이 읽기 쉬운 문자열로 표현한 식별자**이다.  
예를 들어,

```
93.184.216.34 → www.example.com
```

이런 식으로 IP 주소 대신 직관적인 이름을 사용하여  
웹사이트, 서버, 메일 서버 등 인터넷 자원을 식별할 수 있게 한다.

즉, 도메인 이름은 **인터넷에서의 ‘사람 친화적 주소 체계’** 이다.

---

## 2. 구조

도메인 이름은 **계층적(Hierarchical)** 구조를 가진다.  
오른쪽이 상위 계층, 왼쪽이 하위 계층이다.

```
www.example.com
│   │        └── 최상위 도메인 (Top-Level Domain, TLD)
│   └─────────── 2차 도메인 (Second-Level Domain)
└──────────────── 서브도메인 (Subdomain)
```

| 구분                         | 예시                            | 설명                 |
| -------------------------- | ----------------------------- | ------------------ |
| **TLD (Top-Level Domain)** | `.com`, `.org`, `.kr`, `.net` | 도메인 체계의 최상위 계층     |
| **Second-Level Domain**    | `example`                     | 기관, 기업, 개인이 등록한 이름 |
| **Subdomain**              | `www`, `mail`, `blog`         | 세부 서비스나 서버 구분용 이름  |

→ 전체 이름은 오른쪽부터 왼쪽으로 읽으며,  
DNS 조회도 이 계층을 따라 상위 → 하위로 탐색한다.

---

## 3. DNS (Domain Name System)

### (1) 개요

도메인 이름은 사람이 이해하기 쉽지만,
실제 통신은 반드시 **숫자 IP 주소**로 이뤄진다.
따라서 **DNS**는 도메인 이름을 IP 주소로 바꾸는 **분산 데이터베이스 시스템**이다.

즉, DNS는 인터넷의 “전화번호부” 역할을 한다.

---
### (2) 작동 과정

1. 사용자가 브라우저에서 `www.example.com`을 입력한다.
    
2. 운영체제는 로컬 DNS 캐시를 확인하고, 없으면 **DNS 리졸버**에 질의한다.
    
3. 리졸버는 **루트 DNS 서버** → **TLD 서버** → **권한 DNS 서버** 순으로 질의한다.
    
4. 최종적으로 해당 도메인의 IP 주소를 받아 브라우저에 전달한다.
    
5. 브라우저는 IP 주소로 서버에 TCP 연결을 맺는다.
    

이 과정을 **DNS 조회(name resolution)** 라고 부른다.

---

### (3) 캐싱(Caching)과 리졸버(Resolver)

DNS 시스템은 성능과 부하 감소를 위해 **결과를 캐싱(cache)** 한다.

- **로컬 캐시(Local Cache)**
    
    - 운영체제 또는 브라우저가 최근 조회 결과를 저장한다.
        
    - TTL(Time To Live)이 만료될 때까지 재조회 없이 사용한다.
        
    - 예: `/etc/resolv.conf`, 브라우저 내부 캐시
        
- **리졸버 캐시(Resolver Cache)**
    
    - ISP(인터넷 서비스 제공자) 또는 기업 DNS 서버가 관리하는 캐시.
        
    - 다수의 사용자가 동일한 도메인에 접근할 때,  
        반복적인 루트 질의를 피하고 빠르게 응답한다.
        
- **계층적 캐싱 구조**
    
    - 캐시가 없을 경우, 상위 DNS 서버로 순차 질의(Iterative Query).
        
    - 하위 레벨에서 얻은 결과는 상위 계층에 다시 캐싱된다.
        

➡ 캐싱 덕분에 DNS는 전 세계적으로 **지연(latency)을 최소화**하고,  
DNS 서버의 **트래픽 부하를 줄이는 핵심 메커니즘**이 된다.

---

## 4. 주요 구성 요소

|구성요소|설명|
|---|---|
|**루트 네임서버 (Root Name Server)**|전 세계 13개의 루트 서버가 TLD 서버 위치를 알려준다.|
|**TLD 네임서버**|`.com`, `.org`, `.kr` 등 TLD별 도메인 정보를 관리한다.|
|**권한 네임서버 (Authoritative Name Server)**|실제 도메인의 IP 주소(A Record 등)를 저장한다.|
|**리졸버 (Resolver)**|클라이언트 대신 DNS 질의를 수행하는 로컬 네임 서버.|

---

## 5. 레코드 타입

DNS는 단순히 이름-주소만 매핑하지 않는다.  
다양한 정보가 “레코드(Record)” 형태로 저장된다.

|타입|의미|예시|
|---|---|---|
|**A**|도메인 → IPv4 주소|`example.com → 93.184.216.34`|
|**AAAA**|도메인 → IPv6 주소|`example.com → 2001:db8::1`|
|**CNAME**|별칭(Alias) 지정|`blog.example.com → example.com`|
|**MX**|메일 서버 정보|`mail.example.com`|
|**NS**|네임서버 지정|`ns1.example.com`, `ns2.example.com`|
|**TXT**|인증/기타 정보|SPF, DKIM, 도메인 검증 등|

---

## 6. 도메인 이름의 역할

- 사람이 **기억하기 쉬운 주소 체계**를 제공한다.
    
- 네트워크 구조와 **무관하게 서비스 이동 가능** (IP 변경 시에도 이름 유지)
    
- **보안** (TLS 인증서, 도메인 검증, 이메일 인증 등)과 밀접히 연관됨
    
- 전 세계적 표준 시스템 (ICANN이 관리)
    

---

## 7. 동작 예시

사용자가 `https://www.google.com` 접속 시:

1. 브라우저는 로컬 DNS 캐시를 확인
    
2. 없으면 리졸버를 통해 `www.google.com` 질의
    
3. 루트 → `.com` TLD → 구글 권한 서버 순으로 IP 주소를 획득
    
4. 예: `142.250.72.196`
    
5. 브라우저는 이 IP로 TCP 연결을 맺고 [[HTTPS]] 요청 전송
    

---

## 8. 요약

> 도메인 이름은 **IP 주소의 인간 친화적 표현**이며,  
> **DNS**는 이를 실제 네트워크 주소로 변환한다.  
> **리졸버와 캐시 메커니즘** 덕분에 인터넷 전역의 이름 해석이  
> 빠르고 안정적으로 유지된다.
