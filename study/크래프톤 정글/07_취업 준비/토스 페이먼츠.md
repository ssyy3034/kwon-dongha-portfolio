1번. 구현 난이도가 가장 높았던 프로젝트

StoLink 프로젝트에서 PG 연동 크레딧 결제 시스템을 구축하며 데이터 정합성을 확보한 경험입니다.

초기에는 검증, 외부 PG사 API 호출, DB 반영을 하나의 트랜잭션으로 묶어 구현했습니다. 이후 같은 프로젝트에서 외부 API 호출이 트랜잭션 내부에서 무한 대기하며 DB 커넥션 풀을 고갈시키는 Cascading Failure를 겪었고, 결제 플로우에도 동일한 구조적 위험이 있음을 인지했습니다.

이를 계기로 TransactionTemplate을 도입해 검증, 외부 PG사 호출, DB 반영을 3단계로 수동 분리하여 외부 API 대기 구간에서 DB 커넥션을 점유하지 않도록 선제적으로 격리했습니다. 환불 시 PG사 처리는 완료되었으나 로컬 DB 반영이 실패하는 불일치는 보상 로그와 스케줄러로 Eventual Consistency를 보장했고, 동시성 제어는 잔액 차감에 비관적 락, 상태 전이에 낙관적 락을 적용해 자원 특성별로 분리했습니다.

그 결과 외부 PG사 응답 지연이 발생하더라도 커넥션 풀에 영향을 주지 않는 구조를 확보했으며, 100스레드 동시 요청에서 정합성을 안정적으로 유지했습니다.

2번. 가장 효과적으로 트러블슈팅을 했던 사례

StoLink 프로젝트에서 프로덕션 서버가 간헐적으로 모든 API 요청에 응답하지 못하고 멈추는 장애가 발생했습니다. 기능 발표 일정이 있어 우선 컨테이너 재시작으로 임시 대응한 뒤, 쓰레드 덤프를 분석해 원인 추적에 들어갔습니다.

확인 결과, AI 이미지 서버 헬스체크 호출에서 RestTemplate의 기본 타임아웃이 무한 대기로 설정되어 있었습니다. 이 호출이 @TransactionalEventListener 내부에서 REQUIRES_NEW로 별도 트랜잭션을 할당받은 상태에서 실행되고 있어, 외부 서버 무응답 시 쓰레드가 DB 커넥션을 점유한 채 무한정 대기했습니다. 대기 쓰레드가 누적되며 HikariCP 커넥션 풀이 고갈되었고, 정상 요청들까지 커넥션을 얻지 못해 톰캣 쓰레드 풀 전체가 마비되는 연쇄 장애로 이어졌습니다.

RestTemplateBuilder를 통해 5초 타임아웃을 명시 설정하여 외부 장애 시 즉시 예외를 던지고 자원을 반환하도록 수정했습니다. 개선 후 외부 서버 연결을 의도적으로 차단해 검증한 결과, 이전에는 대기 요청 20건만으로 서버 전체가 마비되었으나, 수정 후에는 외부 서버가 완전히 다운된 상황에서도 코어 서비스 응답이 정상 유지되었고, 잦았던 수동 컨테이너 재시작 운영 이슈가 해소되었습니다.
