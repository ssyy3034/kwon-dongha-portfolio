현업 개발자가 자신의 경험을 있는 그대로 진솔하게 풀어내는 느낌으로, 문장을 훨씬 짧고 담백하게 재작성했습니다.

---

**1. 지원 동기**

동시에 쏟아지는 요청 속에서도 데이터 정합성을 지켜내는 서버를 구축하는 데 큰 흥미를 가지고 있습니다. 크래프톤 정글에서 OS 커널을 구현하며 스레드 동기화와 교착 상태를 로우레벨부터 다루었고, 이후 실전 프로젝트에서는 결제 시스템의 잔액 정합성을 비관적 락과 멱등키로 제어하며 동시성 문제에 대한 감각을 익혔습니다.

에스제이에이치스튜디오의 채용 공고를 보며, 수많은 유저의 베팅과 재화 변동을 오차 없이 실시간으로 처리해야 하는 웹보드게임 서버가 제가 가장 고민하고 싶은 문제를 다루고 있다고 느꼈습니다. 실무에서 Scala, Akka, Cassandra를 다뤄본 적은 없습니다. 하지만 부트캠프에서 5개월간 낯선 기술의 동작 원리를 파악하고 즉각적으로 적용하는 훈련을 반복했습니다. Java와 JVM, 동시성에 대한 이해를 바탕으로, 에스제이에이치스튜디오의 개발 환경에 빠르게 적응하여 팀에 기여하고 싶습니다.

**2. 동시성 제어 경험 — Akka 액터 모델을 이해할 수 있는 기반**

Akka의 액터 모델을 직접 사용해 보지는 않았지만, 이전 프로젝트 경험을 통해 그 필요성은 충분히 체감하고 있습니다.

StoLink 프로젝트의 결제 시스템을 구현할 때, 다수의 사용자가 동시에 크레딧을 조작하면 잔액이 어긋나는 문제를 겪었습니다. 비관적 락으로 직렬화하고 멱등키로 중복 결제를 방어해 문제를 해결했지만, 동시에 공유 자원에 대한 락 방식은 트래픽이 커질수록 필연적인 성능 병목을 유발한다는 한계도 확인했습니다.

공유 상태에 락을 거는 대신 각 액터가 독립적인 상태를 갖고 메시지로 소통하는 Akka의 구조는 이러한 락 경합 문제를 근본적으로 해결하는 방식이라고 생각합니다. 스레드 기반 동시성의 원리와 한계를 직접 마주했기 때문에, 액터 모델이 어떤 문제를 해결하고 어디에 주의해야 하는지 빠르게 파악할 수 있다고 생각합니다.

**3. [아키텍처 최적화]

Cassandra를 실무에서 직접 운영해 본 적은 없지만, 데이터 성격에 맞춰 관계형 DB와 NoSQL을 적절히 활용해 본 경험이 이를 빠르게 이해하는 밑거름이 될 것입니다.

과거 프로젝트에서 결제 트랜잭션의 강한 일관성은 PostgreSQL로 보장하고, 스키마가 유연해야 하는 챗봇 대화 로그는 MongoDB에 저장했으며, 휘발성 데이터는 Redis 캐싱으로 분리했습니다. 이 과정을 통해 단일 DB에 모든 것을 담기보다 목적에 맞는 저장소를 선택하는 시야를 길렀습니다.

특히 JPA의 N+1 문제를 해결하며 '데이터를 어떻게 저장하느냐보다 어떤 패턴으로 조회하느냐'에 맞춰 구조를 최적화하는 고민을 깊게 해보았습니다. 카산드라 역시 철저히 쿼리(조회) 중심으로 데이터를 모델링한다고 알고 있는데, 이는 제가 RDBMS에서 조회 성능을 끌어올리기 위해 겪었던 고민의 방향과 매우 비슷하다고 느꼈습니다.

또한 결제 로직에서 비관적 락으로 정합성을 맞추며 트래픽 병목의 한계를 직접 마주해 보았기에, 대규모 트래픽 처리를 위해 일관성을 유연하게 타협하는 카산드라의 접근 방식이 어떤 고민에서 출발했는지 충분히 공감할 수 있었습니다.

이처럼 RDB 환경에서 데이터 정합성과 쿼리 성능을 두고 고민해본 경험들은, 입사 후 카산드라라는 새로운 환경의 설계 철학을 낯설지 않게 받아들이고 빠르게 실무에 적응하는 가장 튼튼한 징검다리가 될 것이라 생각합니다.

**4.[협업과 리더십]**

저는 코드를 잘 짜는 것만큼이나 동료들이 편하게 일할 수 있는 환경을 만드는 데 관심이 많습니다.

5인 팀 프로젝트 당시, 반복되는 Git 커밋과 PR 과정이 팀 전체의 개발 흐름을 끊고 있다는 점을 발견했습니다. 이를 해결하기 위해 Git에 능숙한 팀원과 협력하여 관련 작업을 단일 명령으로 자동화하는 도구를 만들었고, 이 도구는 다른 팀들에게까지 자연스럽게 확산되었습니다.

개발 외적으로도 이와 같은 협업 경험이 있습니다.. 맥도날드에서 약 3년간 팀 리더로 일하며 시프트당 최대 20명의 크루를 관리했습니다. 주문이 몰리는 피크 타임에 병목을 판단해 인력을 재배치하고 현장의 갈등을 중재하며 다양한 성향의 사람들과 호흡을 맞추는 법을 배웠습니다. 뛰어난 개인보다 원활하게 소통하는 팀이 더 좋은 성과를 낸다고 믿습니다. 입사 후에도 동료들과 조화롭게 일하며 함께 성장하는 개발자가 되겠습니다.
