### Part 1. 백엔드 핵심 및 데이터베이스 (깊이 검증)

면접관은 JPA N+1이나 DTO 변환은 신입들이 흔히 쓰는 소재이므로, 그 이상의 **트랜잭션 생명주기나 커넥션 풀에 대한 이해도**를 주로 파고들 것입니다.

1. **[트랜잭션 & 커넥션 풀]** 토스페이먼츠 연동 시 외부 API 호출 지연으로 커넥션 풀 고갈 장애가 발생했다고 하셨습니다. 외부 I/O를 트랜잭션 밖으로 분리하셨다고 했는데, 구체적으로 Spring의 `@Transactional`을 어떻게 제어(또는 분리)하셨나요?
2. **[동시성 제어]** 결제 시스템에 비관적 락(Pessimistic Lock)을 사용하셨네요. 낙관적 락(Optimistic Lock)이나 분산 락(Redis Redisson 등)을 사용할 수도 있었을 텐데 비관적 락을 선택한 이유는 무엇인가요? 데드락(Deadlock)이 발생할 위험은 없었나요?
3. **[멱등성]** 트래픽 폭증 시 중복 결제를 막기 위해 멱등키를 구현하셨다고 했습니다. 이 멱등키는 어디에, 어떤 형태로 저장했나요? 만약 결제 승인 요청 중 서버가 죽는다면 재시도 처리는 어떻게 보장되나요?
4. **[JPA 최적화]** DTO Projection과 Fetch Join을 사용해 API 응답 속도를 18배 개선하셨네요. 둘 중 어떤 병목이 더 컸나요? 컬렉션(`.OneToMany`)에 대해 Fetch Join을 할 때 발생할 수 있는 문제(예: 페이징 불가)는 없었나요?
5. **[캐싱 전략]** Aidiary 프로젝트에서 LLM 응답에 Redis 결과 캐싱을 적용하셨습니다. '자정 TTL'을 설정하셨는데, 만약 밤 11시 59분에 요청한 사용자는 1분 만에 캐시가 만료되나요? 캐시 갱신 전략(예: Cache Stampede 방지)은 고민해보셨나요?

### Part 2. 아키텍처 및 비동기 처리 (설계 역량 검증)

성능 개선 폭이 가장 큰 부분이며(1.16 → 1949 TPS), 면접관이 아키텍처적 사고를 검증하기 딱 좋은 파트입니다.

6. **[비동기 아키텍처]** ML 이미지 합성을 위해 RabbitMQ를 도입하셨습니다. 단순 HTTP 비동기 호출이나 Kafka 대신 RabbitMQ를 선택한 기술적 근거가 무엇인가요?
7. **[메시지 큐]** 이미지 합성 요청이 몰려서 Worker가 감당하지 못하고 큐에 메시지가 계속 쌓이는 백프레셔(Backpressure) 상황이 온다면, 시스템은 어떻게 대응하도록 설계되어 있나요?
8. **[이중 서버 구조]** Spring Boot와 Flask로 서버를 나누셨습니다. 하나의 Python 언어로 백엔드를 통합하거나, Java에서 바로 모델을 호출하지 않고 꼭 이중 구조를 가져가야 했던 이유가 있나요?
9. **[Graph DB]** StoLink에서 PostgreSQL과 Neo4j(Graph DB)를 함께 사용하셨습니다. RDBMS의 재귀 쿼리(Recursive CTE) 만으로는 인물 관계도 조회가 어려웠나요? 두 DB 간의 데이터 정합성은 어떻게 맞췄나요?

### Part 3. AI 워크플로우 & DX (주도성 및 트렌드 검증)

이력서에서 가장 눈에 띄고 흥미로운 파트입니다. 면접관은 '이 지원자가 프롬프트 엔지니어링 수준을 넘어, 진짜 개발 프로세스를 자동화할 줄 아는지' 묻고 싶을 것입니다.

10. **[AI 워크플로우]** 1,000줄의 `CLAUDE.md`를 3-Layer(Core, Appendix, Spec)로 분리하셨습니다. 프롬프트를 청킹(Chunking)하거나 RAG를 붙이는 방식도 있는데, 왜 계층형 문서 분리를 선택하셨나요?
11. **[에이전트 파이프라인]** Architect, Stylist, Auditor, Librarian 에이전트를 나누었다고 했는데, 이 파이프라인은 어떻게 동작하나요? 스크립트 기반 동작인가요, 아니면 AI 자율 에이전트인가요?
12. **[검증 체계]** 에이전트 커스텀 커맨드(Antigravity)를 팀에 확산시켰다고 하셨습니다. AI가 작성한 코드를 바로 PR로 올린다면, 위험한 코드가 main에 머지될 위험은 없나요? 휴먼 인 더 루프(Human-in-the-loop)는 어디에 들어가 있나요?
13. **[문제 해결]** 팀 내에서 '자연스럽게 정착'된 것이 아니라 룰로 제안하셨다고 했죠. 팀원들이 AI 도구 사용을 낯설어하거나 귀찮아하는 등 반발은 없었나요? 어떻게 설득하셨나요?
14. **[LLM 제어]** Aidiary에서 아이 챗봇의 할루시네이션을 막기 위해 LangGraph의 명시적 노드 설계와 3중 가드레일을 썼다고 하셨습니다. 텍스트 기반 프롬프트 제어와 비교했을 때 LangGraph 노드 분리 방식의 장점이 정확히 무엇인가요?

### Part 4. 프론트엔드 최적화 (풀스택 역량 검증)

프론트엔드 작업도 수준급의 수치가 나와 있습니다. JS 동작 원리와 브라우저 렌더링에 대한 이해도를 물어볼 것입니다.

15. **[브라우저 렌더링]** SVG에서 Canvas로 전환하여 INP를 420ms에서 64ms로 줄였다고 했습니다. 브라우저 렌더링 파이프라인 관점에서 SVG와 Canvas의 성능 차이가 왜 발생하나요?
16. **[알고리즘 및 자료구조]** $O(n^2)$ 트리 변환을 HashMap을 써서 로직을 $O(n)$으로 개선하셨네요. 트리가 너무 커지면 HashMap을 만드는 메모리 오버헤드는 문제 되지 않았나요?
17. **[멀티스레딩 in 브라우저]** Canvas 이미지 압축을 위해 Web Worker를 사용하셨습니다. 메인 스레드와 Web Worker 간에 이미지 데이터(`ImageData`나 `Blob`)를 넘길 때 직렬화 비용(Clone) 병목은 없었나요? (힌트: `Transferable Objects`)
18. **[번들 최적화]** Vite `manualChunks`로 초기 번들을 58% 절감했습니다. 어떤 기준으로 청크를 나누셨나요? 혹시 너무 잘게 쪼개서 네트워크 워터폴(Waterfall)이 발생하지는 않았나요?
