#### 서버 다운으로 인한 서비스 장애 해결 - 원인 파악 후 코드 개선

프로덕션 서버가 간헐적으로 모든 API에 응답 불가한 현상이 발생했습니다. CPU·메모리는 정상이었기에 쓰레드 덤프를 분석해보니 다수의 톰캣 쓰레드가 RestTemplate.getForObject()에서 BLOCKED 상태였습니다.

원인을 추적해보니, AI 이미지 서버 헬스체크용 RestTemplate에 Timeout이 @Value로 선언만 되어있고 객체 생성 시에는 적용되지 않아 기본값(무한 대기)으로 동작하고 있었습니다. 또한 이 호출이 @TransactionalEventListener(REQUIRES_NEW) 내부에서 실행되어, 외부 무응답 시 DB 커넥션을 점유한 채 대기 → HikariCP 풀 고갈 → 전체 마비로 이어지는 Cascading Failure였습니다.

근본 원인은 Spring 빈 라이프사이클 이해 부족이었습니다. @Value 필드 주입은 생성자 이후에 이루어지기 때문에, 생성자에서는 Timeout 값(0)이 반영되지 않았습니다. @PostConstruct를 적용해 Spring이 필드 주입을 완료한 직후 Timeout 5초가 실제 적용된 RestTemplate을 초기화하도록 수정했습니다.

##### 크레딧 결제 동시성 제어 및 멱등키

잔액 100에서 50크레딧 결제가 동시에 2건 들어오면, 둘 다 100을 읽고 50을 남기는 갱신 손실이 발생해 데이터 정합성이 깨집니다.또한 네트워크 재시도에 의한 이중 결제 위험도 있었습니다. 비관적 락으로 동시 접근을 직렬화하고, @Version을 통한 낙관적 락을 방어적으로 병행했습니다. 중복 결제는 DB UNIQUE 멱등키 + 토스 Idempotency-Key 헤더로 PG 측까지 이중 방어했습니다.

Testcontainers로 PostgreSQL을 띄워 검증했습니다. Mock DB에서는 락 동작을 신뢰할 수 없기 때문입니다. 100스레드 동시 차감 시나리오를 10회 반복하여 잔액 정합성 100%를 확인했고, 이 테스트가 CI에 포함되어 결제 코드 변경 시 회귀를 자동 감지합니다.

##### 계층형 문서 조회 N+1 해결 - Fetch Join + In-Memory 트리로 450ms -> 25ms 

에디터 문서 사이드바의 로딩이 체감상 느려, 크롬 개발자 도구의 Network 탭을 확인한 결과 사이드바 트리를 호출하는 API의 응답 시간이 450ms를 차지하고 있었습니다. 무거운 비즈니스 로직이 없는 단순 읽기 API였기에 쿼리 로그로 검증한 결과, 문서 30개 조회 시 부모-자식 관계의 Lazy Loading으로 인해 문서마다 부모를 개별 조회하는 N+1 문제가 발생하여 총 61개의 쿼리가 실행되고 있었습니다. @BatchSize도 검토했으나, 사이드바 트리는 항상 전체 데이터가 필요하므로 부분 로딩의 이점이 없다고 판단했습니다.

이에 Fetch Join으로 전체 문서를 1개의 쿼리로 로드한 뒤, 서버에서 HashMap을 활용해 O(N) 시간 복잡도로 트리를 조립하는 방식을 택했습니다. 그 결과 호출 당 쿼리는 61개에서 1개로, 응답 속도는 450ms에서 25ms로 18배 개선되었습니다.

##### AI 컨텍스트 오염 방지 → 3-Layer 문서 및 멀티 에이전트 파이프라인 구축

Claude Code 지침서가 1,000줄을 넘어가면서 단순 텍스트 수정 같은 작업에도 전체 문서가 로딩되어 컨텍스트 윈도우의 약 20%가 규칙 해석에 소모되었고, 레거시 API 명세가 최신 문서와 혼재되어 잘못된 코드가 생성되는 문제가 있었습니다. 또한 협업 과정에서 문서 갱신 누락으로 중복 구현이 발생하는 프로세스 병목이 있었습니다.

이를 해결하기 위해 문서를 3-Layer(Core/Appendix/Spec)로 분리하고, Supervisor 커맨드가 작업 규모(S/M/L)에 맞춰 필요한 layer만 선택적으로 로딩하는 파이프라인을 구축했습니다. S 사이즈 작업 기준 약 50줄 분량만 로딩하도록 경량화하여 유효 토큰 비율을 약 2.5배 확보했고, 단순 작업의 처리 시간을 체감 기준 약 70% 단축했습니다. 레거시 API 명세는 Spec layer에서 분리·아카이빙하여 최신 명세만 참조되도록 했고, 작업 완료 시 Librarian 커맨드가 커밋 기록과 변경 코드를 확인하여 증분 기반으로 문서를 자동 갱신함으로써 문서-코드 간 정합성을 유지했습니다.
