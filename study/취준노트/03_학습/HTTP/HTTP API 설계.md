리소스 : 회원을 등록하고 수정하는게 리소스 X
 - 회원 그 자체가 리소스!!
계층형 구조로 설계해야함.
회원 목록 조회 /members
특정 회원 조회 /members/{id}

URI는 리소스만 식별해야함. 명사로
행위는 어떻게 구분

HTTP 메서드
GET : 리소스 조회
POST : 요청 데이터 처리. 주로 등록에 사용
PUT : 리소스를 대체, 해당 리소스가 없으면 생성
PATCH : 리소스 부분 변경
DELETE : 리소스 삭제

GET
리소스 조회
서버에 전달하고 싶은 데이터는 query를 통해 전달.
바디를 사용하여 전달할수 있지만, 권장 X

POST
요청 데이터 처리
메세지 바디를 통해 서버로 요청 데이터 전달
201로 응답하면 생성된 자원"의" location을 같이 보냄

어쩔수 없는 부분이 있음. URI를 사용한 컨트롤 URI로 동사를 통해 요청 데이터 처리할수도 있음
조회 데이터를 넘겨야 하는데 GET 쓰기 어려울때 POST
- GET 메세지 바디에 데이터 넣는거 지원 안하는 경우가 많아서
단점으로는 POST로 보내면 캐싱이 어렵다.

PUT
데이터 덮어버림
클라이언트가 리소스를 식별 - 리소스 위치를 알아야함!

PATCH
리소스의 부분만 변경함
패치를 못받아들인다? post 사용


안전,멱등,캐시가능

안전 - 호출해도 리소스를 변경하지 않는다.
안전은 해당 리소스 호출만 

멱등
한번 호출하든 100번 호출하든 결과가 똑같다.
post는 멱등 X. 두번 호출하면 중복 발생 가능
활용 - 자동 복구 메커니즘
멱등은 외부 요인으로 중간에 리소스가 변경되는건 고려하지 않음
캐시 가능
