#### 서버 다운으로 인한 서비스 장애 해결 - 원인 파악 후 코드 개선

프로덕션 서버가 간헐적으로 모든 API에 응답 불가한 현상이 발생했습니다. CPU·메모리는 정상이었기에 쓰레드 덤프를 분석한 결과, 다수의 톰캣 쓰레드가 AI 이미지 서버 헬스체크용 RestTemplate.getForObject()에서 BLOCKED 상태였습니다. 

추적해보니 해당 RestTemplate의 Timeout이 @Value로 선언만 되어있고 객체 생성 시에는 적용되지 않아 기본값(무한 대기)으로 동작하고 있었습니다. 이는 @Value 필드 주입이 생성자 이후에 이루어지기 때문에, 생성자 시점에서는 Timeout 값이 0으로 남아있던 Spring 빈 라이프사이클 문제였습니다.

여기에 이 호출이 @TransactionalEventListener(REQUIRES_NEW) 내부에서 실행되고 있어, 외부 서버 무응답 시 DB 커넥션을 점유한 채 무한 대기 → HikariCP 풀 고갈 → 전체 API 마비로 이어지는 Cascading Failure 구조였습니다.

우선 @PostConstruct를 적용해 Spring의 필드 주입 완료 직후 Timeout 5초가 반영된 RestTemplate을 초기화하도록 수정하여 즉각적인 장애는 해소했습니다. 

다만 근본 원인은 트랜잭션 내부에서 네트워크 I/O를 수행하는 아키텍처적 결함이었습니다. 해당 코드는 로직이 강하게 결합되어 있어 즉시 리팩토링하기 어려웠지만, 이 경험을 교훈 삼아 이후 결제 PG 연동 설계에서는 트랜잭션 외부에서 PG사 통신을 완료한 뒤 결과 DB 업데이트에만 최소한의 트랜잭션을 여는 구조를 선제적으로 적용했습니다.

##### 크레딧 결제 동시성 제어 및 멱등키

크레딧 잔액 100에서 50크레딧 결제가 동시에 2건 들어올 경우, 둘 다 100을 읽고 50을 남기는 갱신 손실이 발생하여 데이터 정합성이 깨지는 문제가 있었습니다. 또한 네트워크 재시도에 의한 이중 결제 위험도 존재했습니다.

동시성 문제는 비관적 락으로 동시 접근을 직렬화하되, 브라우저 리다이렉트와 PG 웹훅이 동시에 도달하는 케이스에서는 @Version 낙관적 락 충돌을 감지하여 이미 처리된 요청을 재시도 없이 성공 반환하도록 처리했습니다. 

이중 결제는 DB UNIQUE 제약의 멱등키와 토스 Idempotency-Key 헤더로 클라이언트-PG 양측에서 차단했습니다.

검증은 Testcontainers로 PostgreSQL을 띄워 진행했습니다. 100스레드 동시 차감 시나리오를 10회 반복하여 잔액 정합성 100%를 확인했고, 이 테스트를 CI에 포함하여 결제 코드 변경 시 회귀를 자동 감지하도록 구현했습니다.
##### 계층형 문서 조회 N+1 해결 - Fetch Join + In-Memory 트리로 450ms -> 25ms 

에디터 문서 사이드바의 로딩이 체감상 느려, 크롬 개발자 도구의 Network 탭을 확인한 결과 사이드바 트리를 호출하는 API의 응답 시간이 450ms를 차지하고 있었습니다.

무거운 비즈니스 로직이 없는 단순 읽기 API였기에 쿼리 로그로 검증한 결과, 문서 조회 시 부모-자식 관계의 Lazy Loading으로 인해 문서마다 부모를 개별 조회하는 N+1 문제가 발생하여 총 N+1개의 쿼리가 실행되고 있었습니다.

이에 Fetch Join으로 전체 문서를 1개의 쿼리로 로드한 뒤, 서버에서 HashMap을 활용해 O(N) 시간 복잡도로 트리를 조립하는 방식을 택했습니다. 

@BatchSize도 검토했으나, 사이드바 트리는 항상 전체 데이터가 필요하므로 부분 로딩의 이점이 없다고 판단했습니다.

그 결과 호출 당 쿼리는 N+1개에서 1개로, 응답 속도는 450ms에서 25ms로 18배 개선되었습니다.

##### PG 연동 커넥션 풀 고갈 방지 — I/O-트랜잭션 분리 및 보상 재시도 설계

위 Cascading Failure를 겪은 뒤, 결제 플로우에도 동일한 구조적 위험이 있음을 인지했습니다. 결제 승인 로직이 하나의 트랜잭션 안에서 토스페이먼츠 API를 동기 호출하고 있었고, HikariCP 기본 풀이 10개인 상황에서 PG 응답이 1~2초 소요되므로 동시 결제 10건만으로도 일반 API까지 커넥션 부족이 전파될 수 있는 구조였습니다.

이에 네트워크 I/O와 DB 트랜잭션을 분리했습니다. TX1에서 검증과 상태 전환 후 커밋하여 커넥션을 반환하고, 트랜잭션 밖에서 PG API를 호출한 뒤, TX2에서 승인 결과를 반영하는 구조입니다. 분리로 인해 발생하는 데이터 불일치 엣지 케이스에는 보상 트랜잭션과 재시도 구조로 최종적 일관성을 확보했습니다.

그 결과 PG 응답 지연이 DB 커넥션 풀에 영향을 주지 않는 구조로 전환되었고, 결제 API에 지연이 생겨도 일반 API 정상 동작을 확인했습니다.

##### AI 컨텍스트 오염 방지 → 3-Layer 문서 및 멀티 에이전트 파이프라인 구축

Claude Code 지침서가 1,000줄을 넘어가면서 단순 텍스트 수정 같은 작업에도 전체 문서가 로딩되어 컨텍스트 윈도우의 약 20%가 규칙 해석에 소모되었고, 레거시 API 명세가 최신 문서와 혼재되어 잘못된 코드가 생성되는 문제가 있었습니다. 또한 협업 과정에서 문서 갱신 누락으로 중복 구현이 발생하는 프로세스 병목이 있었습니다.

이를 해결하기 위해 문서를 3-Layer(Core/Appendix/Spec)로 분리하고, Supervisor 커맨드가 작업 규모(S/M/L)에 맞춰 필요한 layer만 선택적으로 로딩하는 파이프라인을 구축했습니다. 

S 사이즈 작업 기준 약 50줄 분량만 로딩하도록 경량화하여 유효 토큰 비율을 약 2.5배 확보했고, 단순 작업의 처리 시간을 체감 기준 약 70% 단축했습니다.

레거시 API 명세는 Spec layer에서 분리·아카이빙하여 최신 명세만 참조되도록 했고, 작업 완료 시 Librarian 커맨드가 커밋 기록과 변경 코드를 확인하여 증분 기반으로 문서를 자동 갱신함으로써 문서-코드 간 정합성을 유지했습니다.
